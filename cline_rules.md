# Cline Rules：定义特定指令与编码标准

**Rules (规则)** 是以 Markdown 格式编写的文件，为 Cline 在所有对话中提供持久的指令。通过定义规则，你可以让 Cline 自动遵循你的偏好，而不必在每次开启新任务时重复说明。

### 规则的适用场景：

- **遵循团队编码标准**（命名规范、文件组织、错误处理模式）。
    
- **理解项目特定上下文**（技术栈、架构决策、依赖关系）。
    
- **应用一致的文档或测试要求**。
    
- **记忆约束条件**，例如“不要修改 `/legacy` 中的文件”或“始终使用 TypeScript”。
    

---

## 1. 支持的规则类型

Cline 具有良好的兼容性，可以自动识别其他主流工具的规则文件：

|**规则类型**|**存放位置**|**描述**|
|---|---|---|
|**Cline Rules**|`.clinerules/`|Cline 的原生主要规则格式|
|**Cursor Rules**|`.cursorrules`|自动检测并兼容|
|**Windsurf Rules**|`.windsurfrules`|自动检测并兼容|
|**AGENTS.md**|`AGENTS.md`|跨工具兼容的标准格式|

所有检测到的规则都会显示在 **Rules 面板** 中，你可以单独切换它们的开启状态。

---

## 2. 规则存储位置

规则可以存储在**项目工作区**或**系统全局**中：

- **工作区规则 (Workspace Rules)**：存放在项目根目录的 `.clinerules/` 文件夹内。适用于团队标准、项目约束以及任何需要通过版本控制（Git）共享的内容。
    
- **全局规则 (Global Rules)**：存放在系统的 Cline Rules 目录中。适用于跨项目的个人偏好。
    

### 目录结构示例：

```Plaintext
your-project/
├── .clinerules/              # 工作区规则
│   ├── coding.md             # 编码标准
│   ├── testing.md            # 测试要求
│   └── architecture.md       # 架构决策
├── src/
└── ...
```

> **注意**：Cline 会合并处理 `.clinerules/` 内所有的 `.md` 和 `.txt` 文件。如果工作区规则与全局规则冲突，**工作区规则优先级更高**。

### 全局规则目录对照表：

|**操作系统**|**默认位置**|
|---|---|
|**Windows**|`Documents\Cline\Rules`|
|**macOS**|`~/Documents/Cline/Rules`|
|**Linux/WSL**|`~/Documents/Cline/Rules` 或 `~/Cline/Rules`|

## 3. 创建与管理规则

### 如何创建：

1. **打开 Rules 菜单**：点击 Cline 面板底部、模型选择器左侧的**天平图标**。
    
2. **创建新文件**：点击 “New rule file...” 并输入文件名（如 `coding-standards`）。
    
3. **编写内容**：使用 Markdown 格式编写指令。
    
    _你也可以使用 `/newrule` 斜杠命令让 Cline 以交互方式为你创建规则。_
    

### 切换开关：

每条规则旁边都有一个开关。这让你能够精细控制哪些规则应用于当前任务。例如，在快速原型开发时，你可以临时关闭严格的测试规则。

## 4. 编写高效规则的技巧

### 结构化布局

规则应具有可读性和针对性。利用 Markdown 的标题和列表：

```Markdown
# 规则标题
(可选) 关于规则存在原因的简要背景。

## 类别 1
- 具体指令
- 带有示例的指令：`像这样编写代码`
- 引用文件：参见 /src/utils/example.ts
```

### 最佳实践：

- **具体而非模糊**：不要只说“使用描述性的变量名”，而应说“变量使用 camelCase，类使用 PascalCase，常量使用 UPPER_SNAKE”。
    
- **包含“为什么”**：解释规则背后的原因，帮助 Cline 在边界情况下做出更好的决策。
    
- **引用示例**：如果代码库中已有成熟模式，直接指向它。例如：“遵循 `/src/utils/errors.ts` 中的错误处理模式”。
    
- **保持更新**：过时的规则会干扰 Cline。如果技术栈变了，请同步更新规则。
    
- **单文件单职责**：按主题拆分文件（如 `testing.md`, `style.md`），方便按需开关。

## 5. 条件触发规则 (Conditional Rules)

条件规则允许你将规则限定在代码库的**特定部分**。规则仅在你处理匹配的文件时激活，从而节省上下文（Context Tokens）并保持焦点。

### 工作原理

通过在规则文件的顶部添加 **YAML Frontmatter** 来实现：



```Markdown
---
paths:
  - "src/components/**"
  - "src/hooks/**"
---
# React 组件准则
当创建或修改 React 组件时：
- 使用函数式组件和 Hooks
- ...
```

### `paths` 条件支持的模式：

- `*`：匹配除 `/` 外的任何字符。
    
- `**`：递归匹配任何字符（包含 `/`）。
    
- `{a,b}`：匹配 a 或 b。
    

|**模式示例**|**匹配范围**|
|---|---|
|`src/**/*.ts`|`src/` 下的所有 TypeScript 文件|
|`**/*.test.ts`|项目中任何位置的测试文件|
|`packages/{web,api}/**`|`web` 或 `api` 包下的所有文件|

---

## 6. 什么算作“当前上下文”？

Cline 根据以下信息评估是否激活条件规则：

1. **你的消息**：提问中提到的路径（如“更新 `src/App.tsx`”）。
    
2. **打开的标签页**：编辑器中当前打开的文件。
    
3. **可见文件**：当前活动窗口中显示的文件。
    
4. **已编辑文件**：任务期间 Cline 创建或修改的文件。

## 7. 实用示例

### 前后端规则分离

``` Markdown
# .clinerules/backend.md
---
paths:
  - "src/api/**"
  - "src/db/**"
---
# 后端准则
- 所有数据库查询必须通过 Repository 层
- 返回类型化的错误，不要直接抛出异常
```

### 文档编写规则



``` Markdown
# .clinerules/docs.md
---
paths:
  - "docs/**"
  - "**/*.md"
---
# 文档准则
- 标题使用 Sentence case
- 每一项功能必须包含代码示例
```

## 8. 故障排除

- **规则未激活**：
    
    - 检查上下文中的文件路径是否与 glob 模式匹配。
        
    - 确保规则在面板中已开启。
        
    - 确认 YAML Frontmatter 的 `---` 分隔符格式正确。
        
- **规则意外激活**：
    
    - 检查 `**` 模式是否过于宽泛。
        
    - 检查是否有匹配该模式的标签页处于打开状态。
        
- **内容直接输出了 YAML 代码**：
    
    - YAML 解析失败。请检查语法错误（如未加引号的特殊字符或缩进问题）。
