
> rule / workflow / skill 本质上都是“对可执行能力的结构化描述”。

但它们在**抽象层级、控制权位置、执行粒度**上是不同的。

我们用系统架构视角拆开。

---

# 一、先给一个直观对比

|概念|本质|控制谁|描述什么|粒度|
|---|---|---|---|---|
|**Rule**|约束条件|约束模型行为|行为边界|最小|
|**Workflow**|过程编排|控制步骤顺序|执行流程|中等|
|**Skill**|能力模块|提供能力接口|可调用功能|最大|

你觉得它们都像 markdown，是因为：

> 它们都是“元语言”，是对系统行为的说明文档。

但在系统结构中，它们处在不同层。

---

# 二、Rule 是什么？

Rule 是：

> **对行为的约束规范**

它回答的问题是：

- 你不能做什么？
    
- 输出必须满足什么格式？
    
- 触发条件是什么？
    

例如：

```
- 禁止输出敏感信息
- 必须用JSON格式返回
- 如果输入包含xxx则拒绝
```

它不是“做事流程”，也不是“能力模块”。

它更像：

> 行为边界定义器。

架构类比：

- 类似防火墙规则
    
- 类似数据库约束
    
- 类似API参数校验
    

Rule 不执行任务，它限制任务。

---

# 三、Workflow 是什么？

Workflow 是：

> **任务执行路径的显式拆解**

它回答的是：

- 先做什么？
    
- 再做什么？
    
- 哪一步依赖哪一步？
    
- 失败如何处理？
    

例如：

```
1. 获取用户输入
2. 调用搜索API
3. 总结搜索结果
4. 输出报告
```

它是“步骤编排”。

架构类比：

- 类似 Airflow DAG
    
- 类似 CI/CD pipeline
    
- 类似状态机
    

Workflow 不定义能力本身，它只组合能力。

---

# 四、Skill 是什么？

Skill 是：

> **一个可被调用的能力单元**

它回答的是：

- 我能做什么？
    
- 我的输入输出格式是什么？
    
- 我能调用哪些工具？
    

例如：

```
Skill: WebSearch
Input: query
Output: search_results
Capability: 调用搜索引擎API
```

它是：

> 能力接口 + 执行逻辑 + 工具绑定

架构类比：

- 微服务
    
- 插件模块
    
- 函数库
    
- 可调用 API
    

Skill 是“功能单元”。

---

# 五、它们在系统中的层级关系

可以抽象成：

```
          Rule
            ↓
        Workflow
            ↓
          Skill
            ↓
          Tool
```

解释：

- Rule 限制 Workflow 如何运行
    
- Workflow 调度 Skill
    
- Skill 调用 Tool（真实API/代码）
    

---

# 六、为什么你觉得它们本质一样？

因为在 LLM 时代：

> 它们都被写成自然语言或 Markdown。

但这只是表达形式。

本质差别在于：

|概念|是“描述”还是“执行”|
|---|---|
|Rule|约束元描述|
|Workflow|结构化流程描述|
|Skill|可执行能力单元|

Markdown 只是：

> 人类可读的配置语言。

---

# 七、更底层一点讲

从计算模型角度看：

- Rule ≈ 约束函数
    
- Workflow ≈ 有向图（DAG）
    
- Skill ≈ 可调用函数（Function）
    

如果抽象成数学表达：

- Rule = f(x) → Boolean（是否合法）
    
- Workflow = Graph(Nodes, Edges)
    
- Skill = g(input) → output
    

---

# 八、为什么现在三者混在一起？

因为 LLM 的出现改变了一件事：

> 逻辑控制从代码转移到 prompt。

以前：

- Rule 是写在程序里的 if 语句
    
- Workflow 是写在 orchestrator 里的 DAG
    
- Skill 是写在代码里的函数
    

现在：

- 这三者可以全部写成自然语言。
    

所以看起来像“都是 markdown 模块”。

但那只是表现形式收敛。

---

# 九、更哲学一点

你隐约感受到的是：

> 这些东西本质上都是“对行为的元建模”。

没错。

Agent 系统本质上是在做：

> 用自然语言构建控制系统。

而不是构建真正意义上的智能。

---

# 十、关键差异总结一句话

- **Rule = 边界**
    
- **Workflow = 路径**
    
- **Skill = 能力**
    

或者更抽象一点：

- Rule 决定“不能越界”
    
- Workflow 决定“怎么走”
    
- Skill 决定“你会什么”
    

---
