# Cline 定制化指南：核心系统概览

了解 **Rules (规则)**、**Skills (技能)**、**Workflows (工作流)**、**Hooks (钩子)** 和 **.clineignore** 如何协同工作来深度定制你的 Cline 体验。

Cline 开箱即用是一款通用的 AI 助手。通过定制化，你可以将其转变为精通你的代码库、团队规范和工作流程的专家。与其在每次任务中重复相同的指令，不如定义一次，让 Cline 自动遵循。

Cline 提供了五个系统来实现这一点，每个系统都有不同的用途和触发时机：

## 核心功能快速对比

|**功能**|**用途**|**激活时机**|**最佳适用场景**|
|---|---|---|---|
|**Rules (规则)**|定义 Cline 的行为准则|始终开启（或根据上下文）|编码标准、项目约束、团队规范|
|**Skills (技能)**|按需加载的领域专家知识|由匹配的请求触发|专业知识、复杂流程、机构特有的经验|
|**Workflows (工作流)**|逐步的任务自动化脚本|通过 `/workflow.md` 调用|重复性过程、发布流程、环境搭建脚本|
|**Hooks (钩子)**|在关键时刻注入自定义逻辑|针对特定事件自动执行|校验、强制执行、监控、自动化触发器|
|**.clineignore**|控制文件访问权限|始终开启|排除依赖项、构建产物、大型数据文件|

---

## 深入理解各工具

### 1. Rules (规则)

**Rules** 是持久化的指导方针。当你希望 Cline 一贯地遵循某些模式时（如编码标准、命名规范、架构约束或特定项目上下文），请使用它。Rules 塑造了 Cline 在所有任务中**如何**工作。

- **示例：** “始终使用 TypeScript” 或 “数据访问层遵循 Repository 模式”。
    

### 2. Skills (技能)

**Skills** 是仅在相关时加载的领域专业知识。当你拥有大量知识但不想在所有场景下都占用上下文空间时，请使用它。Cline 在启动时会读取技能描述，只有当你的请求与之匹配时，才会激活完整的指令。

- **示例：** 一个数据分析技能可能包含 pandas 模式、可视化偏好和输出格式，Cline 仅在处理数据文件时才会加载这些内容。
    

### 3. Workflows (工作流)

**Workflows** 是你按需调用的显式任务脚本。当你有一个需要每次以相同方式运行的可重复多步流程时，请使用它。

- **示例：** 输入 `/release.md`，Cline 就会执行你的发布序列：升级版本、运行测试、更新变更日志、提交、打标签并推送。Workflows 定义了**要做什么**的每一个步骤。
    

### 4. Hooks (钩子)

**Hooks** 是在关键时刻自动运行的程序化护栏。当你需要通过自定义代码来校验、强制执行或扩展 Cline 的行为时，请使用它。

- **示例：** 在 TypeScript 项目中阻止创建 `.js` 文件、在保存前运行 linter（代码检查器），或在部署后通知外部服务。
    

### 5. .clineignore

**.clineignore** 控制 Cline 可以访问哪些文件和目录。使用它来从 Cline 的上下文中排除依赖项、构建产物、生成的文件和大型数据文件。这可以减少 Token 使用量、降低成本，并让 Cline 专注于核心代码。

- **工作原理：** 类似于 `.gitignore`，在项目根目录添加模式，匹配的文件将被自动排除。

## 综合示例：发布流程

看看这五个系统如何协同完成一个新版本的发布：

1. **Rules**：确保 Cline 遵循团队的提交信息格式和版本控制策略。
    
2. **Skills**：提供关于 CI/CD 系统的深度知识，当涉及部署问题时，Cline 会加载这些知识。
    
3. **Workflows**：提供显式的 `/release.md` 序列：升级版本、更新变更日志、打标签、推送。
    
4. **Hooks**：在允许提交前验证测试是否通过，或者检查变更日志是否已真实更新。
    
5. **.clineignore**：将 `node_modules/`、构建产物和生成文件排除在上下文之外，使 Cline 保持专注。

## 存储位置

所有五个系统都支持**全局**和**项目特定**的配置：

|**系统**|**全局位置 (Global)**|**项目位置 (Project)**|
|---|---|---|
|**Rules**|`~/Documents/Cline/Rules/`|`.clinerules/`|
|**Skills**|`~/.cline/skills/`|`.cline/skills/`|
|**Workflows**|`~/Documents/Cline/Workflows/`|`.clinerules/workflows/`|
|**Hooks**|`~/Documents/Cline/Hooks/`|`.clinerules/hooks/`|
|**.clineignore**|N/A|`.clineignore`|

### 何时使用哪种存储方式？

- **优先使用项目存储：** 大多数定制化内容应属于项目目录，因为它们与特定代码库挂钩。团队编码标准、部署工作流和架构约束应与它们描述的代码共存。这也意味着定制化配置会随仓库同步，协作者可以自动获取，且变更可以通过 Pull Request 进行审查。
    
- **个人偏好使用全局存储：** 如果你发现自己向每个项目都添加相同的定制内容，请将其移动到全局存储。你偏好的沟通风格、个人生产力工作流和通用的工具都属于这里。全局定制适用于所有项目，但不会进入版本控制，因此不会影响你的队友。
    

> **注意：** 当名称冲突时，项目特定配置具有优先级（**Skills 除外**，Skills 始终是全局配置优先）。这允许你在需要时为特定项目覆盖全局默认值。

## 安全考虑

定制化功能非常强大，它们会塑造 Cline 编写代码的方式、自动执行命令并影响每一次交互。

- **务必审计：** 在将定制化内容添加到项目或全局配置之前，请仔细审查。
    
- **信任来源：** 仅使用来自你信任的来源的定制化配置。像对待在你的环境中运行的任何代码一样审视这些文件。
    

---

## 最佳实践

- **审查内容：** 了解每个定制化文件的作用及原理。
    
- **验证外部来源：** 从 GitHub 仓库或社区分享获取内容时，检查作者声誉、社区评价以及代码是否名副其实。
    
- **警惕危险命令：**
    
    - 删除文件的 Shell 命令（如 `rm`, `del`）。
        
    - 传输数据的命令（如带有 POST 请求的 `curl`, `wget`）。
        
    - 在项目目录之外的文件操作。
        
    - 修改系统配置的命令。
        
- **版本控制：** 将你的定制化配置纳入版本控制，以便跟踪更改、审查差异并在出错时回滚。
    
- **最小化 Hook 触发：** 创建 Hook 时，使用最严格的事件触发器。如果只需要在提交前运行，就不要在每次文件保存时触发。
